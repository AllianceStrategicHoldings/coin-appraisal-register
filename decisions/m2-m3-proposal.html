How to read this proposal

The May 2026 scope document is substantial. Rather than quote one number for the whole thing, I have gone through it feature by feature and split it into three buckets:

  • Build — Inside our wheelhouse. We will build it as part of this engagement.
  • Build w/ fallback — We will build the feature using the spec's documented fallback path. The "premium" version (typically a paid SDK or specialist integration) is not included.
  • Hand off — Outside our wheelhouse, or compliance/liability work that should sit with a specialist. We are not building these. Recommended to subcontract or defer.

This approach lets you keep the price down, stay on schedule, and avoid paying our rate on work better suited to a specialist.

What we recommend handing off

The five items below are inside the scope document but outside our wheelhouse — either because they require a paid third-party SDK, specialist compliance/security work, or a platform we do not support. For each one, we either build a clean fallback path (so the app still works) or recommend you bring in a specialist.

Hand off 1 — Specialist (mobile / SDK)

Driver's license PDF417 OCR via paid SDK

Reliable PDF417 barcode parsing on iOS Safari requires a commercial SDK such as Microblink BlinkID, Scandit, or Mitek. Those run $200–2,000/month in license fees and need a developer with their specific SDK experience. Our default build covers photo capture (uploaded to Airtable as the audit record) plus manual DL number entry — which complies with the scope's "or manual entry of DL number" wording. If you want true automated capture later, a mobile / SDK specialist is the right person to wire it in.

Hand off 2 — Specialist (security / compliance)

Audit-grade SSN / EIN encrypted vault

The scope requires SSN/EIN to be stored encrypted. Airtable provides platform-level encryption-at-rest, which we will use as the default. If your tax counsel requires field-level customer-managed encryption (typical for IRS-reportable PII), the right home for that field is a separate vault — HashiCorp Vault, AWS Secrets Manager, or a managed service like Stripe Identity. That setup is a separate security engagement (~$1,500–3,000) and should be done by someone whose day job is compliance/security work.

Hand off 3 — Tax / compliance specialist

Actual IRS 1099-B filing automation

The rules for which precious-metals transactions are 1099-B reportable are non-obvious — the threshold is not just "$10,000 cumulative"; it depends on specific items and quantities (e.g. 1,000+oz silver bars, 25+ pieces of pre-1965 90% silver coin, etc.). Our build covers the threshold trigger, the SSN/EIN capture flow, and the flag fields on the deal record. Anything beyond that — the actual filing logic, IRS form generation, classification of which transactions trigger reporting — is tax compliance work and should be handled by a tax-tech specialist or your tax counsel.

Hand off 4 — Specialist (state portal integration, if required)

Custom Texas LE report formats beyond formatted PDF email

The scope wording says "emails a formatted transaction report" — that is what we build. If the receiving department (local PD or Texas DPS) actually requires a different format (CSV, structured XML, upload to a state portal, proprietary format), each of those is a meaningfully different build and the recipient typically dictates exact requirements. That is integration specialist territory, not Make.com generalist territory. Confirm the recipient's actual requirement before this becomes an issue.

Hand off 5 — Native mobile (only if required)

Native iOS push notifications for the manager approval gate

iOS Safari Web Push works for installed PWAs (iOS 16.4+) but Apple throttles delivery aggressively — it is not suitable for sub-second manager approval. Our default architecture is short-interval Airtable polling (5s) on the rep's Pending Approval screen, which is reliable on iOS Safari and meets the scope's functional requirement. If you decide later that you need true native push, you are moving from a PWA to a native iOS app — that is a different platform and should be quoted as a separate native build.
