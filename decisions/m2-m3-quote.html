Critical risks and dependencies

The items below are not me asking for outs — they are the parts of the build where outcome depends on something outside my control or where the spec needs a decision before I can lock the architecture. Flagging them now prevents surprises later.

Risk 1 — API access (M2 blocker)

PCGS CoinFacts and CDN Greysheet API access must be confirmed live before M2 starts

Both APIs are gated. PCGS CoinFacts is selectively granted; CDN Greysheet Dealer+ requires a paid subscription with the API tier enabled. The scope document already requires confirmation of all four API keys (PCGS, CDN, GoldAPI, Apify) before M2 begins — I will not start M2 until all four are confirmed returning data in Make.com. If either PCGS or CDN cannot be obtained, the graded coin lookup feature must be rescoped down to manual cert entry plus the Manual_Price_Override fallback table only.

Risk 2 — Driver's license camera capture

Automated DL OCR in a PWA requires a paid third-party SDK

Standard browser camera APIs cannot reliably parse the PDF417 barcode on US driver's licenses, especially on iOS Safari. True automated DL capture (where the rep scans and the DL number is parsed automatically) requires a commercial SDK — Microblink BlinkID, Scandit, or Mitek typically run $200–$2,000/month in license fees. Without one, "camera capture" reduces to: rep takes a photo of the DL (uploaded to Airtable as the audit record) and types the DL number into a manual field. That fallback is functional and complies with the spec's "or manual entry of DL number" wording, but it is not a hands-free scan. Decision needed before M2 starts.

Risk 3 — SSN / EIN encryption (1099-B)

Airtable is not architected for compliance-grade encrypted PII storage

Airtable provides encryption at rest as part of their platform, but it does not offer field-level customer-managed encryption. If the operator's compliance posture requires audit-grade encryption of SSNs and EINs (typical for IRS-reportable data), Airtable is the wrong store for that field. The options are: (a) accept Airtable's platform-level encryption-at-rest as the documented security boundary, (b) encrypt SSN/EIN client-side before writing to Airtable with a dev-managed key, or (c) route SSN/EIN to a separate vault (HashiCorp Vault, AWS Secrets Manager, or a managed service like Stripe Identity). Options (b) and (c) add scope not included in this quote. Client decision required.

Risk 4 — Real-time manager approval gate

iOS PWA push notifications are less reliable than native push

iOS Safari Web Push only works for installed PWAs (iOS 16.4+) and Apple throttles delivery aggressively. A high-value approval gate that depends on push for sub-second manager response will see occasional delays. The reliable architecture is short-interval polling: the rep's "Pending Approval" screen checks Airtable every 5–10 seconds for status change, and the manager acts from a manager-side screen in the PWA. This will be the default approach unless we discover a reason to layer push on top.

Risk 5 — Texas 48-hour LE report format

Spec lists the required fields but not the required format

The recipient (local PD or Texas DPS) typically dictates the format — it might be a PDF email, a structured CSV, or an upload to a state portal. Each is a meaningfully different build. Default assumption is a formatted PDF email per the scope wording ("emails a formatted transaction report"). If the recipient requires a different format, that is a small adjustment, but it needs confirmation before final delivery.

Risk 6 — Apify eBay scraper reliability

Third-party eBay data is subject to upstream rate-limiting and anti-scraping

The Apify eBay Sold Listings actor scrapes a website that actively defends against scraping. Most lookups will return data; some will return empty or partial results, especially under load. The build will treat "no comps returned" as a valid outcome and the rep falls back to judgment. The 50-runs/day plan minimum should be sufficient for normal floor traffic.

What is not in this quote

Anything that happens inside GHL after the webhook fires (workflow logic, SMS templates, follow-up sequences, opportunity pipeline configuration) is handled separately and is not in scope.
  • Recurring third-party API and infrastructure costs — PCGS, CDN Greysheet, GoldAPI, Apify, Make.com Pro tier, Vercel, Airtable, optional DL OCR SDK, optional SSN vault. These are operator costs, not dev costs.
  • Hardware — iPads, mounts, barcode scanners (if external) are not included.
  • Branding assets — logo, accent color, app name. Build proceeds with placeholders; final assets land via follow-up update.
  • Post-launch new features — the M3 PWA contract includes 3 months of bug-fix support. New features after launch are separate scope.
