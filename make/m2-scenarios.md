# M2 Make.com Scenarios — Build Spec (Supabase backend)

Four scenarios wire the PWA to the live Supabase database
(`https://jsklabrbirsidlnxpgmw.supabase.co`). **Every HTTP request below was
executed against the live project on 2026-07-20 and verified working** — the
Make modules just reproduce these exact calls.

## Shared setup

All Supabase calls are Make **HTTP → Make a request** modules with headers:

| Header | Value |
|---|---|
| `apikey` | service_role key (store once in a Make connection/data store) |
| `Authorization` | `Bearer <service_role key>` |
| `Content-Type` | `application/json` |

Base URL: `https://jsklabrbirsidlnxpgmw.supabase.co/rest/v1`

The service_role key bypasses RLS — these scenarios are the ONLY place it
lives. Never in the app.

App env vars to set in Vercel once webhooks exist:
`VITE_CONFIG_LOAD_URL`, `VITE_DEAL_SUBMIT_URL`, `VITE_CUSTOMER_LOOKUP_URL`,
`VITE_MANAGER_PIN_CHECK_URL`.

## Scenario 1 — `config-load.v2` (replaces Airtable version)

**Trigger:** Custom webhook (POST, no body needed) → app's `VITE_CONFIG_LOAD_URL`

1. **HTTP GET** `{base}/config_coin_types?active=is.true&select=*&order=name`
2. **HTTP GET** `{base}/config_margins?select=category,margin_pct`
3. **HTTP GET** `{base}/reps_master?active=is.true&select=id,name&order=name`
4. **HTTP GET** metals-api latest (existing module from M1 bulk-calc; also
   INSERT each metal into `spot_price_cache` — POST
   `{base}/spot_price_cache` body `{"metal":"gold","price_usd_per_ozt":X}` —
   so the offline cache stays warm)
5. **Webhook response** — JSON, same shape the app already consumes:
   ```json
   {
     "coin_types": [ ...map: id,name,metal_type,priced_by,unit_label,
       face_value,oz_metal_per_unit,mult_circulated,mult_uncirculated,
       mult_slabbed, category: margin_category (lowercased)... ],
     "reps": [ {"id","name"} ],
     "spot": {"gold": USDXAU, "silver": USDXAG, "platinum": USDXPT},
     "margins": [ {"category" (lowercased), "margin_pct"} ]
   }
   ```
   Note: app expects lowercase categories (`silver`), DB stores `Silver` —
   lowercase in the mapping.

## Scenario 2 — `deal-submit` (accepted AND declined deals)

**Trigger:** Custom webhook (POST) → app's `VITE_DEAL_SUBMIT_URL`.
Payload shape: see `DealSubmission` in `web/src/lib/dealSubmit.ts`
(`event_type` is `deal_accepted` or `deal_declined`).

1. **Customer upsert** — HTTP POST
   `{base}/customer_master?on_conflict=phone,dob`
   headers + `Prefer: resolution=merge-duplicates,return=representation`
   body:
   ```json
   {"phone":"{{customer.phone}}","dob":"{{customer.dob}}",
    "name":"{{customer.name}}","zip":"{{customer.zip}}",
    "dl_number":"{{customer.dl_number}}","tcpa_opt_in":{{customer.tcpa_opt_in}}}
   ```
   → returns the customer row; keep `id`. ✅ verified

2. **Deal number** — increment in a Make data store (format `CAR-{{n}}`),
   per spec: generated by Make, not a DB sequence.

3. **Deal insert** — HTTP POST `{base}/deal_log` + `Prefer: return=representation`
   Map: `deal_number`, `customer_id` (from 1), `status` = `Pending`,
   `accepted` true/false by event_type, `accepted_at`/`declined_at` =
   `submitted_at`, `payment_method`, `cash_over_9500_ack`,
   `price_lock_24hr`, `price_lock_expires_at`, totals →
   `total_max_payout`/`total_actual_offer`/`total_negotiation_delta`
   (+`margin_pct_captured` = actual/max), spot →
   `spot_gold`/`spot_silver`/`spot_platinum`, analytics →
   `selling_reason`, `referral_source`, `estimated_collection_age`,
   `competitor_offers_received`, `competitor_offer_amount`, object keys →
   `collection_photo_url`, `dl_photo_url`, `acceptance_photo_url`,
   `signature_image_url`, `offer_letter_url`, `tcpa_opt_in`,
   `operator_id` = 1, `location_id` from Config_Operator default.
   ✅ verified incl. all five analytics fields

4. **Line items** — iterate `lines[]` → HTTP POST `{base}/deal_line_items`
   body: `deal_id`, `source_engine` (`bulk` for calculator lines),
   `description` = name, `quantity`, `grade`, `max_payout`, `actual_offer`,
   `negotiation_delta` (= max − actual). ✅ verified

5. **Fan-out (2.10)** — four independent routes (Make Router), each:
   a. POST `{base}/webhook_deliveries` (`destination`, `event_type`,
      `payload`, `status`: `pending`) ✅ verified
   b. HTTP POST to the destination URL (GHL accepted/declined URL, Zoho,
      fourth URL from Config_Operator) with the full payload
   c. On success: PATCH `{base}/webhook_deliveries?id=eq.{id}` body
      `{"status":"delivered","delivered_at":"now"}`; on error (module
      error handler → Resume): PATCH `{"status":"failed","last_error":...,
      "attempts":1,"next_attempt_at":"+15min"}`
   Routes are independent — one failing must not block the others (use
   error handlers per route, never a scenario-level rollback).
   Note: `destination` enum is `primary_db | ghl | zoho | custom`; the
   primary_db "delivery" is steps 3–4 themselves — log it delivered.

5b. **Zip radius** (operator decision 2026-07-21) — computed here, stored on
   the deal as `customer_zip_radius_miles`. Reference point is dynamic:
   - **Event mode** (deal carries an Event_ID): the event's `venue_zip`
     from `config_events`
   - **Store mode**: the active location's `zip` from `config_locations`

   Steps (after the deal insert, PATCH the deal with the result):
   1. GET `{base}/zip_centroids?zip=eq.{{customer.zip}}&select=lat,lng`
   2. GET `{base}/zip_centroids?zip=eq.{{reference_zip}}&select=lat,lng`
   3. If **either** lookup returns empty (PO-box-only zips like 30301 are
      not ZCTAs), leave the field null — do not error the scenario.
   4. Haversine (miles) as a Make formula, radians = deg × pi/180:
      ```
      3958.8 * 2 * asin(sqrt(
        pow(sin((lat2-lat1)*pi/360), 2) +
        cos(lat1*pi/180) * cos(lat2*pi/180) *
        pow(sin((lng2-lng1)*pi/360), 2)
      ))
      ```
      Round to 1 decimal.
   5. PATCH `{base}/deal_log?id=eq.{{deal_id}}` body
      `{"customer_zip_radius_miles": {{miles}}}`

   Verified against the live `zip_centroids` table (33,791 rows loaded from
   the 2024 Census ZCTA gazetteer): 75201→76102 = 30.8 mi,
   75201→90210 = 1,247.2 mi. Loader: `infra/zips/load-zip-centroids.mjs`.
   Prerequisite config: `config_locations.zip` and `config_events.venue_zip`
   must be filled when the operator provides location/event data at kickoff.

6. **Retry sweep** — separate small scheduled scenario (every 15 min):
   GET `{base}/webhook_deliveries?status=eq.failed&next_attempt_at=lte.now`
   → re-POST each, PATCH delivered/failed+backoff. This + the app-side
   queue = the 2.14 offline story.

7. **Webhook response** `{"ok":true,"deal_number":"CAR-…"}`

## Scenario 3 — `customer-lookup`

**Trigger:** Custom webhook (POST `{phone, dob}`) → `VITE_CUSTOMER_LOOKUP_URL`

1. **One HTTP GET** (FK embedding pulls prior deals in the same call —
   ✅ verified):
   ```
   {base}/customer_master?phone=eq.{{phone}}&dob=eq.{{dob}}
     &select=id,name,zip,dl_number,tcpa_opt_in,
       deal_log(deal_number,accepted_at,total_actual_offer,status)
   ```
2. **Webhook response** (shape from `CustomerLookupResponse` in
   `web/src/api/types.ts`):
   - empty array → `{"matched": false}`
   - row → `{"matched": true, "customer": {...},
     "prior_deals": [{"deal_number","date": accepted_at,"total_offer":
     total_actual_offer,"status"}]}`

## Scenario 4 — `manager-pin-check`

**Trigger:** Custom webhook (POST `{pin, tier}`) → `VITE_MANAGER_PIN_CHECK_URL`

1. Compute `sha256(pin)` (Make **Crypto** module) — `reps_master.pin_hash`
   stores SHA-256 hex of the 4-digit PIN (set when reps are seeded at
   kickoff).
2. **HTTP GET**
   `{base}/reps_master?pin_hash=eq.{{sha256}}&is_manager=is.true&active=is.true&select=id,name,locked_until,failed_attempts`
3. Match & not locked → respond `{"valid": true, "manager_id": id}`;
   no match → `{"valid": false}` (3-strike lockout bookkeeping lands with
   the M3 auth build — PATCH failed_attempts/locked_until here when it does).

## Order of operations for the operator-side hookup

1. Build scenario 1, run once, put its webhook URL in Vercel
   `VITE_CONFIG_LOAD_URL` → redeploy → app now boots from Supabase.
2. Build scenario 2 (+retry sweep), URL → `VITE_DEAL_SUBMIT_URL` → test
   deals start landing in `deal_log` (visible live in Table Editor).
3. Scenarios 3 & 4 → remaining two env vars.
4. GHL/Zoho/fourth destination URLs land in `config_operator` row 1 when
   the client provides them (Section 6 kickoff config).
