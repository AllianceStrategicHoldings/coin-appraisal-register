# Supabase — Primary Database (M2)

System of record for deals, customers, and configuration, per the App Build
Final spec Sections 2.14–2.15 and the M2/M3 SOW. Replaces Airtable as primary
database (decision confirmed with operator 2026-07-14); all workflow logic
stays in Make.com — the database holds data and integrity constraints only.

## Layout

- `migrations/20260714000001_m2_core_schema.sql` — all 15 tables:
  - Config: `config_margins`, `config_coin_types`, `config_locations`,
    `config_events`, `config_operator`, `reps_master`
  - Transactional: `customer_master`, `deal_log`, `deal_line_items`
  - Test mode: `test_deal_log`, `test_deal_line_items` (Section 3.5)
  - Protection layer: `override_codes`, `approval_requests` (2.13, 3.3)
  - Infrastructure: `webhook_deliveries` (2.10 fan-out + offline retry),
    `spot_price_cache` (2.14 offline degradation)
- `migrations/20260714000002_rls.sql` — row level security. Anon key reads
  non-sensitive config only; all writes and sensitive reads go through
  Make.com with the service_role key.
- `seed.sql` — Config_Margins defaults + Config_CoinTypes ported from
  `airtable/seeds/coin_types.csv` (M1 live config).

## Applying

Once the operator's Supabase project exists and we have collaborator access:

```sh
supabase link --project-ref <project-ref>
supabase db push          # applies migrations in order
psql $DB_URL -f seed.sql  # or paste seed.sql into the SQL editor
```

## Conventions

- Standard data types only; no business logic in database formulas
  (spec 2.14). CHECK constraints are data integrity, not logic.
- `deal_number` (CAR-XXXX) is generated by Make, not a DB sequence.
- Rep PINs are stored as hashes (`reps_master.pin_hash`), never plaintext.
- Deal-level field names get reconciled against the operator's kickoff
  field map when it arrives; structural shape is per spec.

Validated locally against Postgres (PGlite): migrations + seed apply clean,
insert chain across all tables works, constraints reject invalid states.
